# 4. Database Model

## projects

```sql
create table projects (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  description text,
  client_name text,
  status text default 'active',
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
```

---

## project_inputs

Stores raw briefs, notes, URLs, and uploaded asset references.

```sql
create table project_inputs (
  id uuid primary key default gen_random_uuid(),
  project_id uuid references projects(id) on delete cascade,
  type text not null,
  content text,
  asset_url text,
  created_at timestamptz default now()
);
```

Types:

```txt
brief
note
url
image
document
```

# 5. Workflow Engine

The workflow is stage-based.

```txt
Brief → Direction → Production → Review → Export
```

Each stage has:

```ts
type WorkflowStage = {
  id: string
  name: 'brief' | 'direction' | 'production' | 'review' | 'export'
  status: 'idle' | 'in_progress' | 'needs_review' | 'complete' | 'failed'
  primaryAction: string
}
```

Stage rules:

```txt
Brief complete → Direction unlocked
Direction selected → Production unlocked
Production outputs exist → Review unlocked
Review complete → Export unlocked
```

# 6. Agent Execution Model

Each agent is a server-side function.

```ts
type AgentRunInput = {
  projectId: string
  stage: WorkflowStageName
  context: ProjectContext
  instruction?: string
}
```

General execution:

```ts
async function runAgent(input) {
  create agent_run pending
  build context
  call OpenAI
  validate structured output
  save output
  update stage
  write activity event
}
```

---

# 7. Context Builder

Do not dump all project data into every prompt.

Build scoped context per task.

```ts
buildProjectContext(projectId, options)
```

Context includes:

```txt
Project metadata
Parsed brief
Selected territory
Relevant previous outputs
Recent feedback
Uploaded asset summaries
```

Context rules:

```txt
Brief Agent → raw inputs only
Direction Agent → brief intelligence
Production Agent → selected territory + brief intelligence
QA Agent → target output + brief + territory
Export Agent → approved outputs only
```

# 9. Structured Outputs

Use strict schemas for all AI output.

Recommended:

```txt
Zod schemas
OpenAI structured output
Server-side validation
Fallback repair step
```

Never store unvalidated freeform AI output as primary data.

# 10. Feedback Loop

Feedback actions:

```txt
make_more_premium
less_generic
more_playful
more_tiktok_native
client_safe
shorter
more_visual
```

Flow:

```txt
User selects output
User applies feedback action
System builds focused context
Agent revises output
New output version saved
Feedback event logged
Activity event created
```

Important:

```txt
Do not overwrite old output.
Increment version.
Keep before/after.
```

```txt

```

# 16. Error Handling

All agent failures should:

```txt
mark agent_run as failed
store error
show retry action
write activity event
keep stage usable
```

User-facing copy:

```txt
“Generation failed. Retry with the same context.”
```

Avoid generic “Something went wrong.”

# 20. Demo Path

The prototype must support this exact walkthrough:

```txt
Create project
Paste messy brief
Generate brief intelligence
Generate territories
Select territory
Generate production system
Review output
Apply “less generic”
Export markdown handoff
```

This is the interview demo path.

---

# 21. Definition of Done

MVP is done when:

```txt
A user can complete the full demo path without manual database edits
All AI outputs are saved and reloadable
Feedback creates a new version
Activity feed reflects actions
Export produces usable markdown
UI matches the intended workflow structure
```
